# Solving Distributed Consistency

In a world of microservices, network failures are inevitable. Idempotency ensures that retrying a request doesn't lead to duplicate side effects (like double-charging a customer).

## The Approach

We use a "Request-ID" based approach stored in Redis with a short TTL. Before processing any request, we check if the ID has already been handled.

```typescript
@Idempotent({
  ttl: 3600, // 1 hour
  keyPrefix: 'payment-capture'
})
async capturePayment(id: string) {
  // Safe to retry!
}
```
