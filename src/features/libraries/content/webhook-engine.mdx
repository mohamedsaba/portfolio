# The Reliability Problem

When building a webhook engine, the biggest challenge isn't sending the HTTP request—it's ensuring that the request is eventually delivered even when the recipient's server is down, without overwhelming your own infrastructure.

## Architecture Highlights

1. **Exponential Backoff**: We use a jittered exponential backoff strategy to prevent "thundering herd" problems.
2. **Idempotency Keys**: Every delivery attempt is tracked with a unique ID to ensure exactly-once processing (where possible).
3. **Dead Letter Queues**: Failed deliveries after 25 attempts are moved to a DLQ for manual inspection.

## Architecture Decision Records (ADR)

### ADR-001: Redis for Persistent Queuing
We chose Redis (via BullMQ) over a traditional SQL-based queue to ensure sub-millisecond enqueue latency and atomic job state transitions.

### ADR-002: Jittered Exponential Backoff
To prevent "thundering herd" issues, we implement randomized jitter on top of exponential backoff, ensuring retries are evenly distributed.
