# The Webhook Delivery Dilemma

If you've ever integrated with Stripe, you know the gold standard of webhooks. They are fast, they are reliable, and if your server is down, they politely retry with an exponential backoff until you recover. 

While studying Stripe's architecture and building my own event-driven systems, I realized that webhook delivery is one of those problems that *looks* trivially simple. "Just make an HTTP POST request when the job finishes." 

I decided to build one from scratch to understand why it's actually hard.

## The Naive Approach and Its Catastrophic Failure

My initial proof-of-concept was exactly what most tutorials suggest. When a job finished, I used `axios` to POST the payload to a registered URL.

It took less than a week of stress testing for the system to collapse under its own weight.

I set up a test endpoint that intentionally "tarpitted" requests—accepting the connection but never responding, keeping the HTTP socket open indefinitely. Because I hadn't enforced strict timeouts, the Node.js event loop quickly became choked with hundreds of hanging requests. Memory spiked, connections maxed out, and the entire background processing pipeline ground to a halt.

I had built a system where a single misbehaving downstream endpoint could take down the entire delivery infrastructure.

## Architecting for Hostile Environments

I went back to the drawing board to build `@mohamedsaba/webhook-engine`. The core philosophy was defensive engineering: assume every target URL is actively trying to destroy your server.

### 1. The Asynchronous Queue

The first architectural change was decoupling the webhook delivery from the business logic. 

Instead of making the HTTP request directly, the application simply pushes a JSON payload to a Redis-backed queue (using BullMQ). This ensures that if a downstream endpoint is slow or offline, it has zero impact on the application's performance. The app pushes the event to Redis in 2 milliseconds and moves on.

### 2. Strict Execution Boundaries

To solve the tarpitting issue, I implemented ruthless timeouts using `AbortController`. 

```typescript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 5000); // 5 seconds max

try {
  const response = await fetch(clientUrl, {
    method: 'POST',
    body: JSON.stringify(payload),
    headers: { 'Content-Type': 'application/json', 'Webhook-Signature': signature },
    signal: controller.signal
  });
  
  if (!response.ok) throw new Error(`HTTP Error: ${response.status}`);
} finally {
  clearTimeout(timeoutId);
}
```

If a downstream endpoint doesn't acknowledge the payload within exactly 5000 milliseconds, the connection is severed. Marked as a failure.

### 3. Exponential Backoff and Jitter

When a webhook fails (due to a 5xx error or a timeout), you cannot simply abandon the payload, nor can you hammer the struggling endpoint every second.

I implemented an exponential backoff strategy: retry after 1 minute, then 5 minutes, then 30 minutes, up to a maximum of 3 days. 

Crucially, I added **Jitter** to the retry calculation. If a downstream server comes back online after an outage, you don't want all 10,000 queued webhooks to hit it at exactly the same millisecond. Adding a randomized jitter (e.g., +/- 15%) smooths out the retry spike, protecting the endpoint from a thundering herd.

### 4. Cryptographic Signatures

Security is paramount. The receiver needs to verify that the webhook actually came from a trusted source and wasn't forged by a malicious third party. 

The engine generates an HMAC SHA-256 signature using a shared secret and attaches it to the `Webhook-Signature` header. The payload is signed *after* serialization to prevent any JSON parsing discrepancies from invalidating the hash.

## The Architect's Takeaway

A production webhook engine isn't an HTTP client; it's a defensive fortress. 

By utilizing queues for isolation, strict timeouts for self-preservation, and jittered exponential backoffs for empathy toward downstream servers, I transformed a fragile `axios.post` into an enterprise-grade delivery mechanism. I didn't just build a feature; I engineered reliability.
