# The Illusion of Rate Limiting

I had deployed a public-facing authentication endpoint for one of my projects. To protect against brute-force attacks, I added a standard rate limiter middleware: maximum 5 requests per minute per IP address.

It worked perfectly in local tests.

Then I started thinking: what actually happens when this runs behind a load balancer with multiple instances? I spun up 8 replicas, pointed a simple script at the auth endpoint, and watched the logs. A single IP had managed to hit the endpoint 40 times in a single minute. 

How was this possible with a limit of 5?

## The Distributed Race Condition

The problem was architectural. With multiple application instances behind a load balancer using round-robin routing, each instance had its own in-memory rate limiter (a basic `Map`).

When 40 rapid-fire requests came in, the load balancer distributed them across all 8 instances. Each individual instance saw about 5 requests, checked its local memory, decided the client was within the limit, and processed the attempt. 

I had created 8 independent rate limiters instead of one global one. 

## Moving to Redis (And Creating a New Problem)

The obvious solution was to move the state out of local memory and into a centralized Redis instance. All instances would check the same Redis key.

I wrote the naive implementation:

```typescript
// NAIVE DISTRIBUTED RATE LIMITER - DO NOT USE
async function isRateLimited(ip: string): Promise<boolean> {
  const currentCount = await redis.get(`rate:${ip}`);
  
  if (currentCount && parseInt(currentCount) >= 5) {
    return true; // Blocked
  }
  
  await redis.incr(`rate:${ip}`);
  if (!currentCount) {
    await redis.expire(`rate:${ip}`, 60); // Expire in 60s
  }
  
  return false; // Allowed
}
```

This solved the multi-instance problem, but it introduced a classic **Time-of-Check to Time-of-Use (TOCTOU)** race condition.

If two instances receive a request at the exact same millisecond, they both query `redis.get`. They both see a count of `4`. They both decide to allow the request. They both call `redis.incr`, pushing the true count to `6`. Requests had leaked through.

## The Power of Atomicity: Enter Lua

To build a truly bulletproof rate limiter, the "check" and the "increment" operations must happen atomically. No other Redis commands can be executed between them.

Redis is single-threaded. This means if you can bundle your logic into a single command, you guarantee atomicity. This is exactly what Redis Lua scripting allows.

I replaced the fragile TypeScript logic with a Lua script that executes entirely within the Redis engine:

```lua
-- Atomic Rate Limiter Script
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call("GET", key)

if current and tonumber(current) >= limit then
  return 1 -- Rate limited (True)
end

redis.call("INCR", key)
if not current then
  redis.call("EXPIRE", key, window)
end

return 0 -- Allowed (False)
```

In the Node.js code, you simply evaluate the script:

```typescript
const isLimited = await redis.eval(luaScript, 1, `rate:${ip}`, 5, 60);
if (isLimited === 1) throw new TooManyRequestsException();
```

Because Redis executes the Lua script atomically, it is mathematically impossible for a race condition to occur, regardless of how many instances are hammering the database simultaneously.

## Moving to Sliding Windows

The fixed-window approach above has a flaw: if an attacker fires 5 requests at 0:59 and 5 requests at 1:01, they get 10 requests in a two-second window. 

To solve this, I evolved the Lua script to use a **Sliding Window Log** algorithm using Redis Sorted Sets (`ZSET`). Each request's timestamp is stored, and the limit is calculated based on the exact rolling 60-second window. Because it's all executed in Lua, it remains perfectly atomic and blisteringly fast.

This entire approach became the foundation for `@mohamedsaba/RateLimiterSDK`.

## The Architect's Takeaway

Building security controls in distributed systems requires paranoid thinking. You must assume race conditions are the rule, not the exception.

By pushing the concurrency management down into the data layer—utilizing the atomic execution guarantees of Redis Lua scripts—I eliminated a catastrophic vulnerability and built a rate limiter that could withstand a flood.
